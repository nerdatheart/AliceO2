1 ## MCStepLogger
2 
3 Detailed debug information about stepping can be directed to standard output using the `LD_PRELOAD` env variable, which "injects" a
4  special logging library (which intercepts some calls) in the executable that follows in the command line.
5 
6 
7 ~~~~~~~{.sh}
8 
9 LD_PRELOAD=path_to/libMCStepLogger.so o2sim -m MCH -n 10
10 ~~~~~~~
11 
12 
13 ~~~~~~~
14 [MCLOGGER:] START FLUSHING ----
15 [STEPLOGGER]: did 28 steps
16 [STEPLOGGER]: transported 1 different tracks
17 [STEPLOGGER]: transported 1 different types
18 [STEPLOGGER]: VolName cave COUNT 23 SECONDARIES 0
19 [STEPLOGGER]: VolName normalPCB1 COUNT 3 SECONDARIES 0
20 [STEPLOGGER]: ----- END OF EVENT ------
21 [FIELDLOGGER]: did 21 steps
22 [FIELDLOGGER]: VolName cave COUNT 20
23 [FIELDLOGGER]: ----- END OF EVENT ------
24 [MCLOGGER:] END FLUSHING ----
25 ~~~~~~~

27 The stepping logger information can also be directed to an output tree for more detailed investigations.
28 Default name is `MCStepLoggerOutput.root` (and can be changed
29 by setting the `MCSTEPLOG_OUTFILE` env variable).
30 
31 
32 ~~~~~~~{.sh}
33 
34 MCSTEPLOG_TTREE=1 LD_PRELOAD=path_to/libMCStepLogger.so o2sim ..
35 ~~~~~~~

37 Finally the logger can use a map file to give names to some logical grouping of volumes. For instance to map all sensitive volumes from a given detector `DET` to a common label `DET`. That label can then be used to query information about the detector steps "as a whole" when using the `StepLoggerTree` output tree.

84 ## MCStepLogAnalysis
85 
86 Information collected and stored in `MCStepLoggerOutput.root` can be further investigated using the excutable `mcStepAnalysis`. This executable is independent of the simulation itself and produces therefore no overhead when running a simulation. 2 commands are so far available (`analyze`, `checkFile`) including useful help message when typing
87 
88 ~~~~~~~{.sh}
89 
90 mcStepAnalysis <command> --help
91 ~~~~~~~
92 
93 ### File formats
94 
95 2 file formats having a standardised structure play a role. On one hand, these are the files produced by the step logging which are the input files for the analysis as explained in the following. On the other hand, each analysis produces an output file containing histograms along with some meta information. Sanity and the type of the file can be checked via
96 
97 ~~~~~~~{.sh}
98 
99 mcStepAnalysis checkFile -f <FileToBeChecked>
100 ~~~~~~~
101 ### Analysing the steps
102 
103 The basic command containing all required parameters is
104 
105 ~~~~~~~{.sh}
106 
107 mcStepAnalysis analyze -f <MCStepLoggerOutputFile> -o <parent/output/dir> -l <label>
108 ~~~~~~~
109 where
110 * `-f <MCStepLoggerOutputFile>` passes the input file produced with the `MCStepLogger` as explained above (default name is `MCStepLoggerOutput.root`)
111 * `-o <parent/output/dir>` provides the top directory for the analysis output (if this does not exist, it is created automatically)
112 * `-l <label>` adds a label, e.g. for plots produced later.
113 
114 A `ROOT` file at `parent/output/dir/MetaAnalysis/Analysis.root` is produced containing all histograms as well as important meta information. Histogram objects are derived from `ROOT`s `TH1` classes.
115 
116 ### Further processing of analysis files
117 
118 Files produced as described before can be investigated further or used to plot the histograms therein. The interface to read these files is the class `AnalysisFile` and histograms can be requested by their names.
119 
120 ~~~~~~~{.c}
121 ++
122 // somthing...
123 #include "MCStepLogger/AnalysisFile.h"
124 // some code
125 AnalysisFile af;
126 af.read("path/to/file.root");
127 // print meta info
128 af.printMetaInfo();
129 // get a histogram by name (program will exit if name not found)
130 TH1& histogram = af.getHistogram("nameOfHistogram");
131 // if you know the underlying object is, e.g. of derived class TH1D and you really need that you can do
132 TH1D& otherHistogramCasted = af.getHistogram<TH1D>("nameOfOtherHistogram");
133 // where the program will safely exit immediately in case the casting was not successful
134 //...
135 // modify histogram or do other things
136 //...
137 //to save changes made to histograms do
138 af.write("path/to/output/file.root");
139 ~~~~~~~
140 ### Additional information about the `MCAnalysisManager`
141 
142 There is the staic method `MCAnalysisManager::Instance()` which returns a reference to a static instance. So always make sure you don't copy it but get the reference in case you want to work with that instance on a global scope, i.e.
143 
144 ~~~~~~~{.c}
145 ++
146 // some code
147 auto& anamgr = MCAnalysisManager::Instance();
148 ~~~~~~~
149 ### Additional information about the analysis objects
150 
151 Histograms which should be written to disk in an analysis are managed by `MCAnalysisFileWrapper` objects. These also make sure that no histogram is created twice. Therefore, all of these histograms should be created like `T* myHisto = MCAnalysis::getHistogram<T>(...)` where the template parameter `T` must be a class deriving from ROOT's `TH1`. It then returns a pointer to the desired object. Managing histograms not on the level of an analysis also enables for requesting histograms from another analysis. In that way one can write a custom analysis for a specific use case but can still ask for e.g. for a histogram from the `BasicMCAnalysis` to derive some additional and more generic information about a simulation run. Hence, never manually delete an object obtained like this.
152 
153 ### Comparing analysis values to reference reference values
154 
155 For the `BasicMCAnalysis` there is a small test suite to compare the obtained values from a simulation run to reference values contained in a JSON file. So far, that is a prototype only caring about the total number of steps and total number of tracks obtained in the simulation. The `JSON` file looks as follows
156 
157 ~~~~~~~{.json}
158 
159 {
160  "analysisName": "BasicMCAnalysis",
161  "nSteps": [absoluteNumberOfSteps, relativeTolerance],
162  "nTracks": [absoluteNumberOfTracks, relativeTolerance]
163 }
164 ~~~~~~~
165 The test is steered via
166 
167 ~~~~~~~{.sh}
168 
169 runTestBasicMCAnalysis -- <path/to/Analysis.root> <reference.json>
170 ~~~~~~~
171 Note, that the test does not know anything about the settings of the simulation run, i.e. there are no information about the primary generator of the transport engine etc. The user has to make sure to apply this coherently.

173 ### Writing/running a custom analysis
174 
175 Although providing already a number of different observables, users might want to add custom observables for their analysis. To do so, a directory for custom analyses has to be created where analysis macros can be provided and loaded at run-time. Note, that only the basic analysis is actually contained in the compiled code. One of the main reasons for that is to enable for a coherent comparison between different points in the git history. However, if you feel like there is an important observable missing, feel free to report that.
176 
177 The logic of adding a custom analysis is very similar to that of `Rivet` and the general workflow should look familiar in any case. Say, your analysis macro directory is `$ANALYSIS_MACROS/` where you have your macro `mySimulationAnalysisc.C` (don't place any other files there since these cannot be read...). A skeleton looks as follows, also containing more information on how and why things are implemented like they are:
178 
179 
180 ~~~~~~~{.c}
181 ++
182 // myMCStepAnalysis.C
183 //
184 // headers you need
185 
186 /* Your analysis class has to derive from the base o2::mcstepanalysis::MCAnalysis. All
187  * methods to be implemented are mentioned below.
188  */
189 class MyMCStepAnalysis : public MCAnalysis
190 {
191  public:
192  /* You don't need a fancy constructor, that's it. The base constructor takes care
193  * of registering this analysis to the global o2::MCStepAnalysis::MCAnalysisManager
194  * object. An MCAnalysis object cannot exist without the MCAnalysisManager.
195  */
196  MySimulationAnalysis()
197  : MCAnalysis("MyMCStepAnalysis")
198  {}
199 
200  /* There are 3 methods covering the analysis: initialize, analyze and finalize.
201  * The first two need to be overriden in any case.
202  * These methods are called by the MCAnalysisManager. Other custom methods
203  * can hence only be used for class-internal purposes only.
204  */
205 
206  /* All histograms used in the analysis must be defined here. This is done using
207  * the method MCAnalysis::getHistogram<T>(...) where the template paramter T has to
208  * be a deriving class of ROOT's TH1. The histogram pointers are the managed
209  * globally for further processing, saving, plotting etc. It is also taken care
210  * of the deletion of the pointers. Do not attempt to delete histograms obtained by
211  * MCAnalysis::getHistogram<T>(...). Histograms are uniquely identified by there
212  * name and an MCAnalysis will stop immediately at the initialisation stage if
213  * two histograms with the same name are detected.
214  */
215  void initialize() override {
216 
217  // monitor calls to the magnetic field per volume
218  histMyFirstObservable = getHistogram<TH1D>("myFirstObservable", 1, 0., 1.);
219  // monitor the steps in the r-z plane
220  histMySecondObservable = getHistogram<TH2D>("mySecondObservable", 1, 0., 1., 2, 0., 2.);
221  // more histograms?! Other things to do?
222 
223  }
224 
225  /* This method is used to extract the step information in order to fill histograms
226  * accordingly.
227  */
228  void analyze(const std::vector<StepInfo>* const steps,
229  const std::vector<MagCallInfo>* const magCalls) override {
230 
231  // loop over mag field calls and match to step ID
232  for(const auto& call : *magCalls) {
233  auto step = steps->operator[](call.stepid);
234  mAnalysisManager->getLookupVolName(step.volId, volName);
235  histMyFirstObservable->Fill(volName.c_str(), 1.);
236  }
237 
238  // loop over steps and get z-position and other things as you like
239  for(const auto& step : *steps) {
240  histMySecondObservable->Fill(step.z, std::sqrt( step.x*step.x + step.y*step.y ));
241  // ...and do more, as you like
242  }
243  }
244 
245  /* the finalyze method can be used to do necessary adjustments like scaling or adding
246  * histograms or whatever must be done to them.
247  */
248  void finalize() override {
249  histMyFirstObservable->Scale( 1. / histMyFirstObservable->GetEntries() );
250  // ...and more...
251  }
252 
253  private:
254  // you might want to provide some useful methods for internal use...
255 
256  private:
257  TH1D* histMyFirstObservable;
258  TH2D* histMySecondObservable;
259  // other histogram pointers and/or members for internal usage
260  };
261 
262 /* The hook to bring your analysis into existence so that the MCAnalysisManager
263  * knows. Don't be scared about the way the pointer of the analysis is created. It is all
264  * taken care of by the MCAnalysisManager since the base MCAnalysis constructor
265  * automatically registeres itself to the MCAnalysisManager. So don't worry about
266  * pointers flying around.
267  */
268 void declareAnalysis() {
269  new MySimulationAnalysis("mySimulationAnalysis");
270 }

272 After having this, you are ready to include this in the analysis run by typing
273 
274 ~~~~~~~{.sh}
275 
276 runMCAnalysis analyze -f <MCStepLoggerOutputFile> -o <parent/output/dir> -l <label> -d $ANALYSIS_MACROS -a mySimulationAnalysis
277 ~~~~~~~
278 where now
279 * `-d $ANALYSIS_MACROS` points the executable to the directory of where your macros are located
280 * `-a mySimulationAnalysis` tells which analysis to load. In case you have more analyses in that directory you want to load, just append the names of all analyses you want to run.
281 The output of the custom analysis is written to `parent/output/dir/mySimulationAnalysis/` and that's it.
