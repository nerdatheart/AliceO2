31 #### How do I report failure for a given Algorithm?
32 
33 Whenever the driver process spots an error message, i.e. an error printed via `LOG(ERROR)` facility, when the driver process quits, it will exit with a exit code of 1. This includes any exception reported by the default exception handler.
34 
35 This comes handy, for example in tests.

37 ### Debugging on your laptop
38 
39 The way the DPL currently works is that the driver executable you launch,
40 will then take care of spawning one device per `DataProcessorSpec` in
41 a separate process. This means that in order to debug your code you need to
42 make sure gdb / lldb are actually debugging the right child process.
43 
44 For `gdb` you can use the `follow-fork-mode` setting. See
45 [here](https://sourceware.org/gdb/onlinedocs/gdb/Forks.html) for the full
46 documentation. This is unfortunately not available in
47 [lldb](https://bugs.llvm.org/show_bug.cgi?id=17972).
48 
49 Alternatively you can start your driver executable with the `-s` / `--stop`
50 command line option which will immediately stop execution of the children after
51 the fork, allowing you to attach to them, e.g. for gdb using:
52 
53 
54 ~~~~~~~{.sh}
55 
56 attach <pid>
57 ~~~~~~~
58 
59 or the `lldb` equivalent:
60 
61 
62 ~~~~~~~{.sh}
63 
64 attach -pid <pid>
65 ~~~~~~~

80 ### Integrating with non-DPL devices
81 
82 Given the Data Processing Layer comes somewhat later in the design of O2, it's possible that you already have some topology of devices which you want to integrate, without having to port them to the DPL itself. Alternatively, your devices might not satisfy the requirements of the Data Processing Layer and therefore require a "raw" `FairMQDevice`, fully customised to your needs. This is fully supported and we provide means to ingest foreign, non-DPL FairMQDevices produced, messages into a DPL workflow. This is done via the help of a "proxy" data processor which connects to the foreign device, receives its inputs, optionally converts them to a format understood by the Data Processing Layer, and then pumps them to the right Data Processor Specs. In order to have such a device in your workflow, you can use the [`specifyExternalFairMQDeviceProxy`][specifyExternalFairMQDeviceProxy] helper to instanciate it. For an example of how to use it you can look at
83 [`Framework/TestWorkflows/src/test_RawDeviceInjector.cxx`][rawDeviceInjectorExample]. The `specifyExternalFairMQDeviceProxy` takes four arguments:
84 
85 
86 ~~~~~~~{.cpp}
87 
88 specifyExternalFairMQDeviceProxy("foreign-source",
89  {outspec},
90  "type=sub,method=connect,address=tcp://localhost:5450,rateLogging=1",
91  o2DataModelAdaptor(outspec, 0, 1)
92  ),
93 ~~~~~~~
94 
95 the first one is the usual `DataProcessorSpec` name, the second one is a list of outputs which we will create from the non-DPL device, the third one is a string to connect to the existing topology and a the fourth one is a function of the kind `o2::framework::InjectorFunction` which does the actual conversion. In this particular case we use the `o2DataModelAdaptor()` helper to create such an translation function since we know that our input is already respecting the O2 Data Model and most of the heavylifing can be done automatically.
96 
97 Sending out the results of a computation can be done in a similar manner. Use
98 
99 
100 ~~~~~~~{.cpp}
101 
102 ConfigParamSpec{"channel-config", VariantType::String, "<channel-configuration>", "Out-of-band channel config"}
103 ~~~~~~~
104 
105 to create an out-of-band channel as specified in `channel-configuration` and hen use the `RawDeviceService` to get the raw FairMQDevice and send data hrough such a channel.

107 ## Customizing workflows creation (WIP)
108 
109 Sometimes it's handy to customise or generalise the workflow creation based on
110 external inputs. For example you might want to change the number of workers for
111 a given task or disable part of the topology if a given detector should not be
112 enabled.
113 
114 This can be done by implementing the function:
115 
116 
117 ~~~~~~~{.cpp}
118 
119 void customize(std::vector<o2::framework::ConfigParamSpec> &workflowOptions)
120 ~~~~~~~
121 
122 **before** including the `Framework/runDataProcessing.h` (this will most likely
123 change in the future). Each ConfigParamSpec will be added to the configuration
124 mechanism (e.g. the command line options) allowing you to modify them. Such options
125 will then be made available at workflow creation time via the `ConfigContext`
126 passed to the `defineDataProcessing` function, using the `ConfigContext::options()`
127 getter.

129 ## Completion policies (WIP)
130 
131 By default the data processing of a given record happens when all the fields of
132 that record are present. So if your Data Processor declares it will consume
133 `TRACKS` and `CLUSTERS`, for any given time interval both need to be produced
134 by some other data processor before the computation declared in yours can happen.
135 
136 Sometimes it's however desirable to customise such a behavior, so that some action
137 on the record can happen even if it's not complete. For example you might want
138 to start computing some quantity as a given message arrives and then complete the
139 computation once the record is complete. This is done by specifying by customising
140 the data processing CompletionPolicy. This can be done using the usual **Customization
141 mechanism** where a:
142 
143 
144 ~~~~~~~{.cpp}
145 
146 void customize(std::vector<CompletionPolicy> &policies);
147 ~~~~~~~
148 
149 function is provided before including `runDataProcessing.h`.
150 
151 Each `CompletionPolicy` requires the user to specify the `matcher` to select
152 which device is affected by it, and a `callback` to decide what action
153 expressed by a `CompletionOp` to take on a given input record.
154 
155 Possible actions include:
156 
157 * `CompletionPolicy::CompletionOp::Consume`: run the data processing callback and
158  mark the available fields in the input as consumed.
159 * `CompletionPolicy::CompletionOp::Process`: run the data processing callback, but do
160  not consume the field, which will be available when the next message for the field
161 * `CompletionPolicy::CompletionOp::Wait`: hold on the record but do not process it yet.
162 * `CompletionPolicy::CompletionOp::Drop`: drop the current available fields from the record.
163 
164 The default completion policy is to consume all inputs when they are all present.
165 
166 When the computation is dispatched with a partially completea `InputRecord`,
167 the user can check for the validity of any of its parts via the `InputRecord::isValid()`
168 API.

170 ## Customizing deployment configuration (WIP)
171 
172 By default every device instanciated by the Data Processing Layer connects to
173 the others using the PUB/SUB paradigm. This might or might not be desiderable
174 for some or even all of the connections. For this reason there is now a way to
175 customise the connections based on the ids of the devices being instanciated.
176 
177 In order to do so, one needs to implement the function
178 
179 
180 ~~~~~~~{.cpp}
181 
182 customize(std::vector<o2::framework::ChannelConfigurationPolicy> &policies)
183 ~~~~~~~
184 
185 **before** including `Framework/runDataProcessing.h` (this will most likely
186 change in the future). You can then extend the policies vector with your own
187 `ChannelConfigurationPolicy`. For each device to device edge, the system will
188 invoke the `ChannelConfigurationPolicy::match` callback with the ids of the
189 producer and of the consumer as arguments. If the callback returns `true`,
190 the `ChannelConfigurationPolicy::modifyInput` and
191 `ChannelConfigurationPolicy::modifyOutput` will be invoked passing the input and
192 output channel associated to the two devices, giving the opportunity to modify
193 the matching channels.

197 ## Lifetime support
198 
199 While initially foreseen in the design, Lifetime for Inputs / Outputs has not
200 yet being implemented correctly. However, once that happens, the following behaviors
201 will be implemented (naming foreseen to change). Once implemented it will be possible
202 to specify the following Lifetime types:
203 
204 * Timeframe: an input that gets processed only once.
205 * Condition: an input that is cache on considered valid for multiple computations,
206  according to its IOV.
207 * Transient: an output which is not actually sent. Useful to use the same mechanism
208  of the Message Passing API to create
209 * QA: an output which once send is also proposed as input to the subsequent computation,
210  allowing for accumulating data (e.g. histograms).
211 * SubTimeframe: an input which gets processed only once which has a
212  granularity of less than a timeframe. Within one computation
213  multiple of these can be created. They get sent as soon as
214  they go out of scope.

216 ## Wildcard support for InputSpec / OutputSpec
217 
218 In order to reduce the amount of code which one has to write to define inputs
219 and outputs, we plan to make the InputSpecs and OutputSpecs as veritable
220 matchers, supporting wildcards. For example if your Algorithm supports
221 processing clusters coming from multiple detectors, it will be possible to
222 specify:
223 
224 
225 ~~~~~~~{.cpp}
226 
227 InputSpec{"*", "CLUSTERS"}
228 ~~~~~~~
229 
230 If the user wants to get both clusters and tracks coming from the same detector,
231 it will be possible to write:
232 
233 
234 ~~~~~~~{.cpp}
235 
236 InputSpec{"*", "CLUSTERS"}, InputSpec{"*", "TRACKS"}
237 ~~~~~~~
238 
239 i.e. the first message which arrives will define the wildcard for all the other input
240 spec in the definition.

242 ### Data flow parallelism
243 
244 Data flow parallelism is simply expressed by tuning the data flow, adding explicitly the parallel data paths, using the appropriate `InputSpec` and `OutputSpec`.
245 
246 E.g.:
247 
248 
249 ~~~~~~~{.cpp}
250 
251 DataProcessorSpec{
252  "tpc_processor_1",
253  Inputs{},
254  Outputs{{"TPC", "CLUSTERS", SubSpec(0)}},
255  // ...
256 },
257 DataProcessorSpec{
258  "tpc_processor_2",
259  Inputs{},
260  Outputs{{"TPC", "CLUSTERS", SubSpec(1)}},
261  // ...
262 }
263 // ...
264 DataProcessorSpec{
265  "tpc_processor_18",
266  Inputs{},
267  Outputs{{"TPC", "CLUSTERS", SubSpec(17)}},
268  // ...
269 }
270 ~~~~~~~
271 
272 or alternatively the parallel workflows part could be generated programmatically:
273 
274 
275 ~~~~~~~{.cpp}
276 
277 parallel(
278  DataProcessorSpec{
279  "tpc_processor",
280  {InputSpec{"c", "TPC", "CLUSTERS"}}
281  },
282  18,
283  [](DataProcessorSpec &spec, size_t idx) {
284  spec.outputs[0].subSpec = idx; // Each of the 18 DataProcessorSpecs should have a different subSpec
285  }
286 )
287 ~~~~~~~
288 
289 Similarly this can be done for a component that merges inputs from multiple parallel devices, this time by modifying programmatically the `Inputs`:
290 
291 
292 ~~~~~~~{.cpp}
293 
294 // ...
295 DataProcessorSpec{
296  "merger",
297  mergeInputs({"a", "TST", "A", 0, Lifetime::Timeframe},
298  4,
299  [](InputSpec &input, size_t index) {
300  input.subSpec = index;
301  }
302  ),
303  {},
304  AlgorithmSpec{[](InitContext &setup) {
305  return [](ProcessingContext &ctx) {
306  // Create a single output.
307  LOG(DEBUG) << "Invoked" << std::endl;
308  };
309 }
310 // ...
311 ~~~~~~~
312 
313 When one declares a parallel set of devices you can retrieve the rank (i.e. parallel id) or the number of parallel devices by using the `ParalleContext`, which can be retrieved from the `ServiceRegistry` (see also the `Services` section below), e.g.:
314 
315 
316 ~~~~~~~{.cpp}
317 
318 size_t whoAmI = services.get<ParallelContext>().index1D();
319 size_t howManyAreWe = services.get<ParallelContext>().index1DSize();
320 ~~~~~~~

322 ### Time pipelining
323 
324 A second type of parallelism is time based pipelining. This assumes that the data can be subdivided in subsequent "time periods" that are independent one from the other and which are each identified by some timestamp entity. In this particular case it could result handy that some part of the workflow are actually processing different time periods. This can be expressed via the `timePipeline`, directive, e.g.:
325 
326 
327 ~~~~~~~{.cpp}
328 
329 // ...
330 timePipeline(DataProcessorSpec{
331  "processor",
332  {InputSpec{"a", "TST", "A"}},
333  {OutputSpec{"TST", "B"}},
334  AlgorithmSpec{[](ProcessingContext &ctx) {
335  };
336  }
337 }, 2);
338 // ...
339 ~~~~~~~
340 
341 which will result in two devices, one for even time periods, the other one for
342 odd timeperiods.
