1 # DPL workflows for the TPC
2 
3 ## TPC reconstruction workflow
4 The TPC reconstruction workflow starts from the TPC digits, the *clusterer* reconstructs clusters. Those are
5 *converted* to RAW pages and passed onto the *decoder* providing the decoded (native) cluster format to the
6 *tracker*.
7 
8 The workflow consists of the following DPL processors:
9 
10 * `digit-reader` -> using tool [o2::framework::RootTreeReader](../../../Framework/Utils/include/Utils/RootTreeReader.h)
11 * `clusterer` -> interfaces [o2::TPC::HwClusterer](../reconstruction/include/TPCReconstruction/HwClusterer.h)
12 * `converter` -> implements conversions to raw pages, to be moved to a worker class
13 * `decoder` -> interfaces [o2::TPC::HardwareClusterDecoder](reconstruction/include/TPCReconstruction/HardwareClusterDecoder.h)
14 * `tracker` -> interfaces [o2::TPC::TPCCATracking](reconstruction/include/TPCReconstruction/TPCCATracking.h)
15 * `writer` -> implements simple writing to ROOT file
16 
17 Clusters are created and processed together with the MC labels, however track labels are not yet written
18 to output file. This will come soon.
19 
20 ### Input data
21 The input can be created by running the simulation (`o2sim`) and the digitizer workflow (`digitizer-workflow`).
22 The digitizer workflow produces the file `tpcdigits.root` by default, data is stored in separated branches for
23 all sectors.
24 
25 ### Quickstart running the reconstruction workflow
26 The workflow is implemented in the `tpc-reco-workflow` executable.
27 
28 Display all options
29 ```
30 tpc-reco-workflow --help
31 ```
32 
33 Important options for the `digit-reader` as initial publisher
34 ```
35 --infile arg Name of the input file
36 --treename arg (=o2sim) Name of the input tree
37 --digitbranch arg (=TPCDigit) Digit branch
38 --mcbranch arg (=TPCDigitMCTruth) MC info branch
39 --tpc-sectors arg (=0-35) TPC sector range, e.g. 5-7,8,9
40 ```
41 
42 Options for the `tpc-track-writer` process
43 ```
44 --outfile arg (=tpctracks.root) Name of the input file
45 --treename arg (=o2sim) Name of output tree
46 --track-branch-name arg (=TPCTracks) Branch name for TPC tracks
47 --trackmc-branch-name arg (=TPCTracksMCTruth) Branch name for TPC track mc labels
48 ```
49 
50 Examples:
51 ```
52 tpc-reco-workflow --infile tpcdigits.root --tpc-sectors 0-15
53 ```
54 
55 ```
56 tpc-reco-workflow --infile tpcdigits.root --tpc-sectors 0-15 --disable-mc 1
57 ```
58 
59 ### Global workflow options:
60 ```
61 --input-type arg (=digits) digits, clusters, raw
62 --output-type arg (=tracks) clusters, raw, tracks
63 --disable-mc arg (=0) disable sending of MC information
64 --tpc-lanes arg (=1) number of parallel lanes up to the tracker
65 ```
66 Support for all other output types than `tracks` is going to be implemented soon, multiple outputs
67 will be supported in order to keep the data at intermediate steps.
68 
69 #### Parallel processing
70 Parallel processing is controlled by the option `--tpc-lanes n`. The digit reader will fan out to n processing
71 lanes, each with clusterer, converter and decoder. The tracker will fan in from the parallel lanes.
72

75 #### TPC CA tracker
76 The [tracker spec](src/CATrackerSpec.cxx) interfaces the [o2::TPC::TPCCATracking](reconstruction/include/TPCReconstruction/TPCCATracking.h) worker class which can be initialized using an option string. The processor spec defines the option `--tracker-option`. Currently, the tracker should be run with options
77 ```
78 --tracker-option "refX=83 cont"
79 ```
80 
81 The most important tracker options are:
82 ```
83 cont activation of continuous mode
84 refX= reference x coordinate, tracker tries to propagate all track to the reference
85 bz= magnetic field
86 ```
87 
88 ### Current limitations
89 * track labels not yet written to file
90 * only the full workflow with input type `digits` and output type `tracks` has been tested so far
91 * the propagation of MC labels goes together with multiple rearrangements and thus copy
92 * only one timeframe is processed, this matches the current working scheme of the digitizer workflow
93 * sequential workflow where the TPC sectors are processed individually and the data is buffered in the
94  tracker until complete. We are currently (ab)using DPLs time slice concept to process individual
95  TPC sectors. This is probably anyhow solved sufficiently when the trigger timing is implemented in
96  DPL
97 * raw pages are using RawDataHeader version 2 with 4 64bit words, will be converted to version 3 soon
98 
99 ## Open questions
100 * how to implement further workflow control with multiple timeframes
101 * clarify the buffering of input in the tracker
102 * clarify whether or not to call `finishProcessing` of the clusterer
103 * the tracker reports about clusters being dropped, can be an indication that some conversion step in
104  the workflow is incorrect.
